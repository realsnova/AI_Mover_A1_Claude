# 第 1 部：產品規劃（完整規格書）

## 1.0 重要前提：IP 使用聲明

本專案直接使用寶可夢素材（角色名稱、形象、音效風格）。法律限制（開發與部署必須遵守）：

- 寶可夢為 The Pokémon Company / 任天堂之註冊商標與著作物。本專案**僅限個人學習、家庭或單一課堂內非商業使用**。
- **不得**公開部署到公開網址、上架、收費、放廣告，或於社群大量散布。
- 素材來源使用 PokeAPI（https://pokeapi.co）提供的公開 sprite 圖檔，僅供開發測試用途。
- 所有角色資料集中於 `monsters.json`，日後若需公開上線可整批替換為原創怪獸而不動遊戲邏輯。

## 1.1 核心學習循環（每 3–5 分鐘一輪）
```
遇見野生寶可夢 → 單字題挑戰（聽/看/選/拼） → 答對收服/獲得經驗
→ 寶可夢升級進化 → 解鎖新地圖區域（新單字主題） → 挑戰道館（單元總測驗）
```

## 1.2 成功指標
- 單次使用 15–20 分鐘內完成一個主題關卡
- 每個單字至少以 3 種題型出現（間隔重複）
- 學生不需家長協助即可獨立操作

## 1.4 題型設計（對應官方考試能力，全部做成遊戲互動）

1. **看圖選字（A）**（Reading Part 1 簡化）：出現圖片/emoji，從 3 個選項點選正確單字 → 攻擊命中
2. **聽音選圖（B）**（Listening Part 4 簡化）：播放單字/短句發音（Web Speech API），點選正確圖片
3. **拼字組合（C）**：把打散的字母方塊依序點擊拼出單字 → 投出精靈球收服
4. **句子填空（D）**：例句挖空，三選一 → 對戰技能升級
5. **快問快答**（道館 Boss 戰）：混合以上題型限時作答

## 1.5 遊戲化系統

### 寶可夢機制
- **收服**：每個主題區配置 6–8 隻寶可夢（用 PokeAPI sprite）。答對「拼字組合」題即收服，存入圖鑑。
- **進化**：同一單字在不同天答對 3 次（間隔重複）→ 該區寶可夢進化一階。進化 = 複習的視覺回饋。
- **對戰**：野怪戰答錯不懲罰扣血，只顯示「Miss!」並立刻給正解重試一次（低挫折設計）；道館戰才計分。
- **圖鑑**：收集頁 = 學習進度頁。每隻寶可夢卡片背面就是它綁定的單字卡（字、圖、音、例句）。

### 勇者鬥惡龍機制
- **像素風大地圖**：10 個區域用步行點選移動，DQ 式開場「勇者啊，去收集單字的力量吧！」
- **升級音效與儀式感**：過關播放 DQ 風格升級旋律（8-bit 合成音，避免直接使用原版音源）
- **金幣商店**：答題賺金幣，購買裝飾道具（帽子、背景）——只買外觀，不買能力，避免 pay-to-skip 學習

### 激勵與節奏
- 每日登入獎勵（連續登入給稀有寶可夢遭遇）
- 學習節奏：一次野外遭遇 = 5 個新字 + 5 個舊字混合，符合工作記憶容量
- 錯題自動進入佇列，優先在下次遭遇出現

## 1.6 UI/UX 準則（針對國小三年級電腦操作能力）

三年級學生特性：滑鼠點擊熟練、拖曳不穩定、打字很慢、英打幾乎不會、閱讀中文為主。

**必須遵守：**
1. **一律點選優先**：所有題型都可用「純點擊」完成；拼字題用點擊字母方塊，不強制拖曳、不要求鍵盤輸入
2. **大目標區**：所有可點擊元素 ≥ 64×64px，間距 ≥ 16px
3. **介面中文、學習內容英文**：按鈕、說明、回饋用繁體中文；單字、例句、語音為英文
4. **零文字依賴導航**：主要導航用圖示 + 顏色（圖鑑 = 紅色球、地圖 = 綠色、商店 = 金幣）
5. **即時語音回饋**：每題答完自動朗讀正確單字（SpeechSynthesis, en-US, rate 0.85）
6. **不出現失敗畫面**：答錯只有「再試一次！」，永遠可重試；避免倒數壓力（僅 Boss 戰有寬鬆計時）
7. **單一畫面單一任務**：一個畫面只有一個主要行動，無多層選單
8. **自動存檔**：進度自動保存，無帳號、無登入流程（localStorage）
9. **響應式**：支援學校電腦 1366×768 與平板觸控

## 1.7 技術架構
- **形式**：純前端單頁應用，**單一 HTML 檔**（HTML + CSS + Vanilla JS），無後端、無資料庫
- **語音**：Web Speech API（SpeechSynthesis），零成本免 API key
- **圖片**：寶可夢 sprite 取自 PokeAPI 靜態 CDN；單字圖用 emoji
- **音效**：8-bit 風格用 Web Audio API 程式合成（避免版權音源）
- **存檔**：localStorage
- **資料分離**：wordlist、monsters 與遊戲引擎邏輯分離（但最終需內嵌進 HTML，見第 2 部）

# 第 2 部：技術決策與開發環境（踩坑經驗，務必先讀）

## 2.1 關鍵技術決策（不要走回頭路）

1. **Twemoji CDN**：cdnjs 的 twemoji 是 **404**！正確引用：
   - script：`https://cdn.jsdelivr.net/npm/@twemoji/api@15.1.0/dist/twemoji.min.js`
   - 圖檔 base：`https://cdn.jsdelivr.net/gh/jdecked/twemoji@15.1.0/assets/`（folder: svg, ext: .svg）
2. **emoji 政策**：資料中只放 Windows 10 可顯示的 emoji（Unicode ≤12）；無合適 emoji 的字留空字串 → 遊戲以白底中文字卡顯示（zh-fallback）。Twemoji 是加值層，離線退回原生 emoji 也不能破版。
3. **寶可夢 sprite**：`https://raw.githubusercontent.com/PokeAPI/sprites/master/sprites/pokemon/{id}.png`（shiny 加 `/shiny/`）；img onerror 全域備援顯示 👾。
4. **音效**：全部 Web Audio 程式合成（tone/sfxCorrect/sfxWrong/sfxCoin/jingleVictory），無外部音檔、無版權疑慮。
5. **發音**：SpeechSynthesis en-US rate 0.85；Chrome 需使用者點擊過頁面後才允許（實際使用必然滿足）。
6. **單一檔案原則**：所有 CSS/JS/資料內嵌單一 HTML 檔，使用者雙擊即玩（file:// 環境，**fetch 不可用**）。
7. **版權**：寶可夢素材僅限個人/課堂非商業使用，不得公開部署。

## 2.2 開發環境已知問題與解法（會踩坑）

1. **file:// 下 fetch 讀不到本地 json** → 資料必須內嵌進 HTML（Phase 4 正式內嵌全量資料）。
2. **瀏覽器擴充不能直接開 file:// 網址**。實測注入法（已驗證可行）：
   - navigate 到 `https://example.com` → 用 JS 建立 `<input type="file">` + FileReader 存到 window 變數 → 上傳 HTML 檔 → `document.open(); document.write(window.__HTML); document.close()`
   - 注入後 localStorage/語音/CDN 都正常，可完整測試。
3. **截圖座標 ≠ 視窗座標**：截圖可能縮放。點擊一律用頁面元素 ref，不要用截圖座標。
4. **長流程腳本有超時限制**：自動答題迴圈要拆成多次呼叫（每次 ≤ 3 題）；超時不代表頁面壞了，下次呼叫讀狀態即可。
5. **speechSynthesis 測試**：先真實點擊頁面一次，再測 utterance 的 onstart/onend；直接腳本呼叫會得到 not-allowed（是政策不是 bug）。
6. **document.write 注入後 DOMContentLoaded 可能不觸發**：Twemoji 初始化要用 `document.readyState` 判斷，任何載入方式都會初始化（勿依賴 DOMContentLoaded）。

# 第 4 部：Phase 3 設計與驗證清單（SRS / 進化 / 每日獎勵）

## 4.1 目標
把「玩過就忘」變成「科學複習」：答對的字隔天再考、答錯的字優先重考；複習成果用寶可夢進化視覺化；每日登入獎勵建立使用習慣。

## 4.2 資料結構變更

`state.wordStats[id]` 從 `{correct, wrong}` 擴充為：
```js
{ correct: 0, wrong: 0, box: 1, lastSeen: "", correctDays: [] }
// box: Leitner 盒 1~3（1=生字, 2=熟悉, 3=精熟）
// lastSeen: 最後出題日期 "YYYY-MM-DD"
// correctDays: 答對的「不同日期」清單（最多存 3 筆即可）
```

`state` 新增：
```js
lastLogin: "",      // 最後登入日期
streak: 0,          // 連續登入天數
shinyCaught: []     // 收服的「稀有(異色)」寶可夢 id
```

`state.caught[i]` 新增 `evoStage: 0`。

相容性：`load()` 時用 migrate 函式補齊舊存檔缺少的欄位（頂層用 Object.assign，wordStats 內層需逐筆補 box:1）。

## 4.3 SRS 出題邏輯（Leitner 3 盒）
- 答對 → `box = Math.min(box+1, 3)`；記錄當天到 `correctDays`（不重複）
- 答錯 → `box = 1`
- 到期規則：box1 隨時可考；box2 需 `lastSeen` ≥ 1 天前；box3 需 ≥ 3 天前

# 第 5 部：Phase 4 設計與驗證清單（全量 395 字 / 10 區域 / 劇情 / 儀表板）

## 5.2 資料導入方式（重要）
瀏覽器以 file:// 開啟時 **fetch 讀不到本地 json**，資料必須「內嵌」進 HTML：
- 把 `wordlist.json` 的 `words` 陣列內嵌為 `const WORDS_ALL = [...]`（約 110KB，總檔案 ~160KB，可接受）
- 把 `monsters.json` 內嵌為 `const MONSTERS = {...}`
- 建立查表：`const WORD_EMOJI = {}, WORD_ZH = {}; WORDS_ALL.forEach(w => { WORD_EMOJI[w.word]=w.emoji; WORD_ZH[w.word]=w.zh; })` → 取代 Phase1/2 的 EMOJI_BANK/ZH_BANK（干擾字都是表內單字，直接查表）

## 5.3 區域系統
- `REGIONS` 每區增加欄位：`topic`（對應 wordlist topic）、`monsters`（來自 MONSTERS）、`boss`
- 區域單字：`wordsOf(regionKey) = WORDS_ALL.filter(w => w.topic === region.topic)`
- **解鎖規則**：第 1 區開放；第 N 區需要第 N-1 區徽章；地圖鎖定卡顯示「需要 XX 徽章」
- 收服進度、圖鑑、道館、學習模式全部改為「per region」：
  - `state.caught[i]` 增加 `region` 欄位（migrate：舊資料補 "animals"）
  - 圖鑑頁改為分區顯示（區名標題 + 該區 6-8 格）
  - 道館門檻：該區收服 ≥ 6 隻（7 隻區域為 ≥ 5，時光神殿 7 隻）
- **冠軍之路（mixed，125 字）特別規則**：字量大，內部分 3 章各約 42 字（依 id 順序切），野戰選章出題；道館（超夢）需三章各答對過 30 字才開放

## 5.4 題型調整（配合全量單字）
- 拼字題（C）：僅限 `/^[a-z]{3,8}$/`（含空格、斜線、大寫、連字號的字自動排除，如 bus stop、Monday、stomach-ache、o'clock）
- 聽音題（B）：需要正解有 emoji 且至少 2 個干擾字有 emoji，否則不出 B
- 填空題（D）：挖空邏輯處理大小寫與變形（例 "Bats"→"＿＿＿s"）；多字詞（bus stop）直接整詞挖空
- 看圖題（A）：無 emoji 的字用中文字卡（機制已存在）
